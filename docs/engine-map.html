Vue d'ensemble

Une capture entre dans l'inbox et attend la consolidation batchée (SYN-93) : le scheduler déclenche un cycle sur tout le lot à une heure planifiée (minuit/midi) ou via une valve de taille. Le Step 1 classifie via Claude Haiku 4.5 → 4 sorties non-exclusives possibles (atomic_note, project_entries[N], entities/facts, relations). Les facts passent par un score de confiance qui les répartit en 3 buckets. Les entités peuvent déclencher une proposition de merge si elles ressemblent à une existante (SYN-39). Les project_entries déclenchent une synthèse vivante par projet (append ou refinement si ≥ 20 entries).

Légende des liens

1:N · un parent, plusieurs enfants (ex : 1 inbox → N entities via provenance)
N:M · plusieurs des deux côtés (ex : atomic_notes ↔ entities via la liste de noms)
1:1 · couplage strict (ex : 1 atomic_note → 1 ligne embedding)
Trait pointillé · soft-link réversible ou logique (mentions par nom, merge, promotion depuis une file)

Pattern provenance (SYN-41)

Toutes les projections (entities, facts, relations, atomic_notes, project_entries) gardent un provenance_capture_id qui pointe vers la ligne inbox immuable qui les a spawnées. C'est le squelette de la traçabilité dans l'app (chip "🔗 source #N").

Soft-delete vs hard-delete

Aucune table ne fait de DELETE hors des files temporaires (pending_facts résolus, review_queue). Les disparitions sont des soft-links : merged_into_id sur entities (SYN-39), bientôt archived_at / obsoleted_at sur facts + entities (SYN-37 + SYN-59). Lineage toujours rebuildable.

L'idée

La carte n'est pas une table de plus : c'est une projection assemblée à la demande depuis le graphe existant (entités + notes + relations). Aucune nouvelle source de vérité : juste deux caches (positions, labels). Un recalcul complet ne perd rien.
